# Reflection – Practical 4: Connecting TikTok to PostgreSQL with Prisma ORM

## Documentation: Main Concepts Applied

### 1. PostgreSQL Database Setup
I applied knowledge of relational database administration by creating a dedicated database (`tiktok_db`) and a restricted application user (`tiktok_user`) with the minimum required privileges. This follows the principle of least privilege — the application only has access to its own database and not the entire PostgreSQL instance.

### 2. Prisma ORM
I used Prisma as the data access layer between the Node.js application and PostgreSQL. The key concepts applied were:

- **Schema definition** — Describing data models (User, Video, Comment, Like, Follow) and their relationships directly in `schema.prisma` using Prisma's declarative syntax.
- **Migrations** — Using `npx prisma migrate dev` to translate the schema into SQL DDL statements and apply them to the database, providing version-controlled schema changes.
- **Prisma Client** — Using the auto-generated, type-safe client to perform CRUD operations, relational queries, and aggregations without writing raw SQL.

### 3. Password Hashing with bcrypt
I applied the concept that passwords must never be stored in plain text. The `bcrypt` library was used to hash passwords with a salt before saving them. During login, `bcrypt.compare()` checks the submitted password against the stored hash.

### 4. JWT-Based Authentication
I implemented stateless authentication using JSON Web Tokens. On successful login, the server signs a token containing the user's ID using a secret key. The token is then sent by the client in the `Authorization: Bearer <token>` header for subsequent requests. The `protect` middleware verifies the token's signature and validity on every protected route.

### 5. RESTful API with Protected Routes
I applied middleware chaining in Express, attaching the `protect` middleware to routes that require authentication. This separates authentication logic from controller logic cleanly.

##  Reflection

### What I Learned

This practical gave me hands-on experience with a full backend integration that goes beyond prototyping. The biggest shift was moving from in-memory JavaScript arrays to a real relational database. This made me appreciate how much Prisma simplifies what would otherwise involve writing extensive SQL, managing connection pools, and handling type mismatches manually.

I also gained a much clearer understanding of **why** bcrypt is used instead of simpler hashing algorithms like MD5 or SHA-256. bcrypt is intentionally slow (the cost factor is configurable), which makes brute-force attacks computationally expensive. This is a security design choice, not a limitation.

Working with JWTs helped me understand the difference between **stateful** authentication (server keeps session records) and **stateless** authentication (the token itself carries the claims). Stateless auth scales better because any server instance can verify the token without shared session storage.

The seed script was also valuable — it demonstrated how to create realistic relational test data programmatically, which is far more efficient than entering data manually through Postman.

### Challenges Faced and How I Overcame Them

**Challenge 1: DATABASE_URL format errors**  
When first configuring the `.env` file, Prisma threw a connection error because the `?schema=public` parameter was missing from the connection string. I resolved this by carefully reading the Prisma documentation and correcting the URL format to:
```
postgresql://tiktok_user:your_password@localhost:5432/tiktok_db?schema=public
```

**Challenge 2: Prisma Client out of sync after schema changes**  
After modifying `schema.prisma`, I encountered runtime errors where the Prisma Client methods did not reflect the updated schema. I learned that the client must be regenerated by re-running:

npx prisma migrate dev --name <migration_name>

or at minimum `npx prisma generate` if only the client needs updating.

**Challenge 3: Understanding Prisma relation syntax**  
Defining many-to-many relationships (e.g., users liking videos) was initially confusing. Prisma requires explicit relation fields and a join model. Reviewing the Prisma docs on implicit vs explicit many-to-many relations clarified the pattern.

**Challenge 4: JWT token not being accepted on protected routes**  
During Postman testing, protected routes kept returning 401 errors even with a valid token. The issue was that the token was being sent without the `Bearer ` prefix in the Authorization header. Once the correct format (`Bearer <token>`) was used, the middleware accepted it.
